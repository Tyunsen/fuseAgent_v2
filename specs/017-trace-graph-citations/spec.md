# Feature Specification: Trace Graph Citations

**Feature Branch**: `[017-trace-graph-citations]`  
**Created**: 2026-04-05  
**Status**: Draft  
**Input**: User description: "1.时间甘特图需要按真实事件和真实时间展开，不能全部挤在一天，不能出现时间结论1/2这类错误标签，而且要画得更清晰易读；2.引用数字必须出现在正文实际引用的位置，不能作为回复卡片右上角的独立[1][2][3][4][5]列表，这些引用数字要可以点击并自动展开右侧参考文档来源抽屉并定位对应序号；3.实体脉络模式在用户明确@知识库且知识库已有大量节点和边时，必须生成对应的知识图谱子图，不能只显示Knowledge Graph和空结果提示。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 时间脉络主图可读且可信 (Priority: P1)

作为在时间脉络模式下阅读回答的用户，我希望主图用真实事件名称和真实时间位置来展示时间线，而不是把所有事情挤在同一天、或显示“时间结论1/2”这种内部标签，这样我才能直接理解事件先后、密度和阶段变化。

**Why this priority**: 时间脉络模式的核心价值就是“按时间整理并直观呈现”。如果主图的时间分布和标签都不可信，这个模式对用户来说就是失效的。

**Independent Test**: 在时间脉络模式下询问“3月发生了什么？”，确认主图为甘特图，任务标签是事件名，且不同日期/时间段会落在不同时间位置，不会全部重叠在一天。

**Acceptance Scenarios**:

1. **Given** 时间脉络回答引用了多个发生在不同日期或不同时间段的事件，**When** 主图渲染完成，**Then** 甘特图必须把这些事件展开到各自不同的时间位置，而不是全部堆叠在同一天。
2. **Given** 时间脉络主图已经生成，**When** 用户查看图中任务名称，**Then** 标签必须直接显示事件名称或清晰事件短语，而不能显示“时间结论1”“时间结论2”等内部编号。
3. **Given** 时间脉络主图已经生成，**When** 用户查看主图区域，**Then** 图形整体必须保持可读，不得因为同日事件重叠、标签过长或布局挤压导致主要信息不可辨认。

---

### User Story 2 - 引用标号出现在正文实际引用处 (Priority: P1)

作为需要核验证据的用户，我希望引用数字直接出现在正文被引用的句子后面，而不是出现在回复卡片的右上角形成独立的 `[1][2][3][4][5]` 列表，并且这些数字可以点击后自动打开右侧“参考文档来源”抽屉并定位到对应来源。

**Why this priority**: 对用户来说，引用的意义在于“知道这句话来自哪里”。如果数字不出现在具体句子位置，而只是漂浮在卡片角落，用户无法建立句子和证据之间的映射。

**Independent Test**: 对一条带来源的回答，检查正文中具体句子后出现 `[n]` 标号；点击任一标号后，右侧来源抽屉自动打开并聚焦到对应编号的来源条目。

**Acceptance Scenarios**:

1. **Given** 回答正文包含来源支撑的结论，**When** 回答渲染完成，**Then** 每处被引用的正文内容后都应出现对应的 `[n]` 标号，而不是只在卡片右上角显示统一的数字列表。
2. **Given** 用户点击正文中的某个 `[n]` 标号，**When** 交互触发后，**Then** 系统必须打开右侧“参考文档来源”抽屉，并自动定位到相同编号的来源条目。
3. **Given** 一条回答没有实际引用来源，**When** 回答渲染完成，**Then** 系统不得显示伪造的 `[n]` 标号，也不得显示误导性的独立数字列表。

---

### User Story 3 - 实体脉络必须生成真实子图 (Priority: P1)

作为在实体脉络模式下查询知识库的用户，当我明确 `@` 指定了知识库，并且知识库本身已经有大量节点和边时，我希望系统一定能生成与本次回答相关的知识图谱子图，而不是只显示“Knowledge Graph”和“当前引用段落没有匹配到可展示的图谱元素”之类的空结果提示。

**Why this priority**: 实体脉络模式的核心交付物就是“围绕问题生成一个可看的实体关系子图”。如果在图谱已经建好的前提下还经常返回空子图，这个模式就无法使用。

**Independent Test**: 在实体脉络模式下，对一个已经建好图谱且节点/边数量充足的知识库发问“说一下现在各方的诉求利益”，确认系统展示出与本次回答相关的实体和边组成的子图，而不是空提示。

**Acceptance Scenarios**:

1. **Given** 用户在实体脉络模式下明确 `@` 指定一个已完成图索引的知识库，**When** 回答生成完成，**Then** 系统必须展示一张与该回答内容相关的知识图谱子图。
2. **Given** 知识库整体图谱已经满足可用门槛且回答本身使用了该知识库来源，**When** 实体脉络主图渲染时，**Then** 系统不得直接降级为空结果提示，除非能证明当前回答确实没有任何可映射的实体和边。
3. **Given** 实体脉络主图已经生成，**When** 用户查看子图内容，**Then** 子图中应只包含与本次回答相关的实体和边，而不是无关的大图全量内容。

---

### Edge Cases

- 当多个时间事件发生在同一天时，系统如何在不失真时间位置的前提下保持任务标签可读？
- 当时间证据只精确到“某月”或“某时间段”时，系统如何避免伪造具体到某一天的时间定位？
- 当正文同一句话同时引用多个来源时，系统如何展示多个引用标号而不破坏阅读流畅性？
- 当用户点击 `[n]` 标号时，如果来源抽屉当前已打开并定位在其他来源上，系统如何平滑跳转到新的目标来源？
- 当实体脉络检索的首轮引用没有直接映射到图元素，但知识库整体图谱明确存在相关节点与边时，系统如何避免过早降级为空图提示？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统必须在时间脉络模式下以甘特图作为唯一主图，并基于真实事件和真实时间信息组织任务位置。
- **FR-002**: 系统必须优先使用真实事件名、事件短语或回答中的可读结论短语作为甘特图任务标签，而不能使用“时间结论1/2/3”等内部编号。
- **FR-003**: 当回答包含多个不同日期或时间段的事件时，系统必须把这些事件展开到各自不同的时间位置，而不能默认压缩到同一天。
- **FR-004**: 当证据只提供月级或区间级时间时，系统必须保留该粒度，不得伪造不存在的日级日期。
- **FR-005**: 时间脉络主图必须保持可读，不得出现因标签重叠、布局挤压或无意义重复导致用户无法辨认主要事件。
- **FR-006**: 系统必须把引用标号放在正文实际被引用的位置，而不是作为回复卡片右上角的独立数字列表。
- **FR-007**: 每个正文引用标号必须采用 `[n]` 形式，并与右侧“参考文档来源”抽屉中的来源条目编号保持一致。
- **FR-008**: 用户点击正文中的 `[n]` 引用标号时，系统必须自动打开右侧“参考文档来源”抽屉并聚焦到对应编号的来源条目。
- **FR-009**: 当一条回答没有实际引用来源时，系统不得显示伪造的 `[n]` 标号，也不得显示脱离正文的独立数字引用列表。
- **FR-010**: 在实体脉络模式下，若用户明确 `@` 指定了一个图索引已完成且整体图谱规模达标的知识库，系统必须优先生成与本次回答相关的知识图谱子图，而不是直接降级为空图提示。
- **FR-011**: 实体脉络模式在生成子图时必须只保留与本次回答相关的实体和边，避免返回无关的大图全量内容。
- **FR-012**: 当实体脉络模式首轮映射失败但知识库图谱整体可用时，系统必须尝试扩展或放宽回答相关图元素匹配，而不是立即返回空结果提示。
- **FR-013**: 系统不得让时间脉络、空间脉络、实体脉络模式因为错误的前端渲染或错误的后端映射而退化回与模式目标不一致的展示。

### Key Entities *(include if feature involves data)*

- **时间脉络事件项**: 时间模式主图中的单个任务单元，包含事件标签、时间位置和关联来源。
- **正文引用标号**: 出现在回答正文中的 `[n]` 标记，用于把具体句子映射到对应来源。
- **来源抽屉条目**: 右侧“参考文档来源”抽屉中的单条来源记录，包含编号、文档名、位置和片段。
- **实体脉络子图**: 围绕当前回答生成的知识图谱子图，只保留相关实体和边。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在 `E:\codes\fuseAgent_v2\iw_docs` 验收数据集上，针对时间脉络问题“3月发生了什么？”，甘特图中可见任务标签 0 条允许使用“时间结论1/2/3”之类的内部编号。
- **SC-002**: 在同一时间脉络验收问题下，只要证据中存在至少 3 个不同的日期或时间段，主图中也必须呈现至少 3 个不同的时间落点或区段，而不是全部挤在同一天。
- **SC-003**: 在带引用的回答中，正文必须出现至少 1 个 `[n]` 引用标号，并且每个标号都能通过点击打开右侧来源抽屉并定位到同编号来源。
- **SC-004**: 在带引用的回答中，回复卡片右上角不得再出现脱离正文的独立 `[1][2][3][4][5]` 数字列表。
- **SC-005**: 在实体脉络模式下，对一个图索引已完成且整体图谱规模达标的 `@知识库` 发问时，主图必须返回一张非空的知识图谱子图，而不是空结果提示。
- **SC-006**: 按 Spec Kit 宪章要求，使用 `iw_docs` 全量导入后，向量、全文与图索引总耗时必须不超过 4 分钟，且图谱节点数大于 80、关系数大于 100。

## Assumptions

- 现有时间、空间、实体三种脉络模式会继续复用当前 QA 与 trace-support 链路，而不是新建独立系统。
- 右侧“参考文档来源”抽屉仍是回答级交互，不升级为全局页面组件。
- 正文引用标号按当前回答内的来源顺序稳定生成，不要求跨不同回答共享同一编号体系。
- 实体脉络空图问题的修复范围限定在“已建好图谱且当前回答确实有相关来源”的场景，不要求对完全无图或完全无关问题强行造图。
