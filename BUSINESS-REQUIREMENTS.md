# fuseAgent 业务需求说明

## 1. 产品定位

fuseAgent 是一个面向内部业务用户的知识库问答系统。

它的核心目标不是让最终用户理解不同引擎、不同建图方式或不同检索方式，而是让用户可以直接完成以下事情：

- 创建和管理知识库
- 上传业务文档并形成可问答的知识库
- 针对知识库提问并获得可信答案
- 查看答案对应的原文证据
- 查看与本次答案相关的知识图谱

最终交付的是一套稳定可用的“问答 + 证据 + 图谱”系统，而不是一个面向最终用户的引擎对比平台。

## 2. 目标用户

- 内部业务人员
- 内部研究人员
- 内部知识库维护人员

v1 阶段默认按内部使用场景设计，登录进入系统的用户都按管理员能力使用，不区分普通用户和管理员。

## 3. 核心业务目标

系统需要做到：

- 用户提出问题后，系统给出一个明确答案
- 这个答案必须尽可能有证据支撑
- 用户可以直接看到证据来自哪份文档、哪一段内容
- 用户可以同时看到与本次回答相关的小型知识图谱
- 当证据不足时，系统必须明确说明“当前证据不足”，而不是强行回答

## 4. v1 范围

### 4.1 数据范围

v1 先支持以下文档类型：

- PDF
- Word
- Markdown
- TXT

v1 不以图片理解或视觉索引为重点。

### 4.2 规模范围

v1 以“几百篇文档量级的知识库”作为主要目标场景，实际可承载规模取决于服务器资源。

### 4.3 场景范围

v1 聚焦于：

- 单知识库问答
- 文档证据追踪
- 关联图谱查看
- 知识库管理

不要求 v1 做成复杂的多角色、多租户、外部开放平台。

## 5. 核心业务对象

### 5.1 知识库

- 系统中最核心的业务对象是知识库
- 一个工作区对应一个知识库
- 用户可以看到知识库列表
- 用户可以新建知识库
- 用户可以进入某个知识库查看文档、管理内容并发起问答

### 5.2 会话

- 每个知识库下可以有多个会话
- 会话用于承载围绕该知识库的连续问答
- 用户可以新建会话、切换会话、查看历史会话
- 会话列表应该清晰可见，便于回到之前的问题上下文

### 5.3 文档

- 文档属于某个知识库
- 用户可以上传、查看、删除、替换文档
- 文档上传后应进入处理流程，最终用于问答、证据和图谱生成

## 6. 用户核心流程

### 6.1 知识库建立流程

用户进入系统后，可以：

1. 新建知识库
2. 设置知识库基本信息
3. 上传文档
4. 等待系统处理完成
5. 进入该知识库开始问答

### 6.2 问答流程

用户在知识库中发起问答时，可以：

1. 创建一个新会话或进入已有会话
2. 在输入框中提问
3. 必要时通过 `@` 的方式关联知识库
4. 获取答案、证据卡片和关联小图谱
5. 继续追问，形成连续会话

### 6.3 证据核验流程

用户看到回答后，可以：

1. 展开证据卡片
2. 查看证据对应的文档名
3. 查看原文段落
4. 查看页码或标题定位信息
5. 点开证据后在当前页面内预览原文并定位到对应片段

### 6.4 图谱查看流程

用户看到回答后，可以：

1. 查看本次问答对应的小图谱
2. 点击图谱中的节点或关系查看详情
3. 通过图谱理解答案涉及的实体和关系
4. 让图谱与证据形成联动

## 7. 功能需求

### 7.1 知识库管理

系统需要支持：

- 展示知识库列表
- 新建知识库
- 查看知识库基本信息
- 进入知识库文档页面
- 进入知识库问答页面
- 展示知识库当前状态

知识库管理页应该清晰、直接、偏工作台风格，不应让用户先理解复杂的系统结构后才能开始使用。

### 7.2 文档管理

系统需要支持：

- 上传文档
- 查看已上传文档
- 看到文档处理状态
- 删除文档
- 替换文档
- 手动触发重建或重跑处理流程

上传文档后，系统默认自动开始处理；同时保留手动重跑入口。

### 7.3 文档处理状态

系统需要清楚展示文档和知识库的处理状态。

要求包括：

- 用户能知道知识库是否已可用于问答
- 用户能知道文档是否还在处理中
- 用户能知道后台是否正在更新
- 首次建库未完成前，不允许提问
- 后续增量更新时，可以继续提问，但要明确提示结果可能滞后

### 7.4 问答结果

每次问答结果至少包括：

- 回答正文
- 证据卡片
- 关联小图谱

这三部分应作为一次完整回答的组成部分出现，而不是彼此割裂的独立模块。

### 7.5 证据卡片

每条证据卡片至少包含：

- 文档名
- 原文段落
- 页码或标题定位信息

证据卡片需要：

- 可以折叠和展开
- 让用户快速判断答案是否可信
- 支持在页面内打开原文预览
- 后续支持跳到原文具体位置

### 7.6 原文预览

用户点击证据后，应在当前页面内打开侧边抽屉式预览，而不是跳离当前问答页面。

预览中应尽量定位到：

- 对应页码
- 对应标题
- 对应段落

### 7.7 图谱展示

图谱的业务目标是帮助用户理解“本次回答关联到了哪些实体与关系”。

因此系统需要：

- 在每次回答中展示本次问答的小图谱
- 图谱应有较好的可视化质量
- 节点、边、背景、交互都应足够清晰
- 用户点击节点或边时，可以看到详细信息

图谱不应只是摆设，而应服务于理解答案和追踪证据。

### 7.8 图谱与证据联动

当用户点击图谱中的节点或关系时，系统应同时支持：

- 高亮相关证据卡片或回答片段
- 打开节点详情或关系详情

也就是说，图谱和证据要能双向联动。

### 7.9 会话能力

系统需要保留会话上下文，但不能让历史上下文无限制干扰当前检索。

业务要求是：

- 回答可以参考会话历史
- 检索以当前问题为主
- 必要时可以带少量上一轮实体或关键词做补充

这意味着系统应具备“连续对话感”，但不能因为上下文累积而让结果越来越失焦。

### 7.10 证据不足处理

当系统没有足够证据支撑回答时：

- 必须直接说明当前证据不足
- 不得编造内容
- 不得伪装成“有依据的回答”

这是系统可信度要求中的关键部分。

## 8. 交互与界面要求

### 8.1 整体界面风格

整体界面应偏简洁、清晰、企业知识库产品风格。

要求包括：

- 知识库列表清晰
- 会话列表清晰
- 聊天区域简洁
- 页面层级明确
- 用户进入系统后应能直接开始干活，而不是先看介绍页

界面参考要求也需要明确固定下来：

- 系统整体 UI 风格以 ApeRAG 为主要参考
- 知识库管理页、会话页、聊天页、输入区、证据卡片等整体产品界面，应尽量贴近 ApeRAG 的清晰、简洁风格
- 图谱相关 UI 风格以 MiroFish 为主要参考
- 节点、边、背景、图谱工具栏、图例、详情面板等与图谱相关的可视化效果，应尽量贴近 MiroFish 的呈现方式
- 用户提供的两个前端原型属于产品体验参考，应尽量在页面排布和视觉层级上贴近原型效果

### 8.2 页面结构

问答页面的核心结构应是：

- 左侧知识库列表
- 左侧会话列表
- 中间问答内容区

一次回答中的图谱放在会话内容中间，与回答和证据卡片放在同一个回答单元里，不再固定放在页面左右两侧。

### 8.3 会话输入区

输入区应当始终清晰可见，不应被图谱、浮层或其他界面元素遮挡。

### 8.4 知识库首页

知识库首页应是一个真正的知识库管理入口，至少要让用户能直接完成：

- 看知识库
- 搜知识库
- 新建知识库
- 进入文档管理
- 进入问答

知识库首页的整体观感与交互组织，也应优先参考 ApeRAG 风格和用户提供的原型排布，而不是临时自由发挥。

## 9. 数据一致性要求

### 9.1 删除或替换文档

当文档被删除或替换后：

- 页面上应立刻标明该文档已经删除或替换
- 该文档不应再继续作为新的证据来源展示
- 底层清理和重建可以异步进行

### 9.2 证据追踪一致性

系统中的证据追踪必须尽量保持一致：

- 用户看到的答案
- 用户看到的证据卡片
- 用户看到的原文预览
- 用户看到的关联小图谱

这些内容之间应尽量互相对应，避免出现明显脱节。

## 10. 产品边界

### 10.1 面向最终用户的交付边界

最终业务用户面对的是一个“知识库问答产品”，不是一个“多引擎试验台”。

因此：

- 用户不需要理解底层引擎区别
- 用户不需要在问答时自己选择引擎
- 用户只需要看到清晰、可信、可追踪的结果

### 10.2 内部评测边界

系统内部可以做建图和检索能力对比，但这属于研发和选型过程，不属于最终业务用户的主要使用流程。

## 11. 一句话总结

fuseAgent 的业务目标是：

打造一套面向内部业务用户的知识库问答系统，让用户能够围绕自己的知识库进行连续问答，并在每次回答中同时看到可信答案、可追溯的段落级证据，以及帮助理解答案的关联知识图谱。

## 12. 当前方案约定

在不展开具体实现细节的前提下，当前已经明确的技术方向是：

- 抽取与建图能力以 MiroFish 为核心
- 问答检索能力以 LightRAG 为主
- 最终交付给业务用户的仍然是一套统一的知识库问答系统，而不是让用户感知底层技术切换
- 系统整体产品 UI 以 ApeRAG 风格为主
- 图谱渲染 UI 以 MiroFish 风格为主
